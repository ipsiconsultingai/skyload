현재 브랜치의 PR에 남겨진 리뷰를 분석하고 적용합니다.

## 실행 순서

1. 현재 브랜치의 PR 번호 확인: `gh pr view --json number -q .number`
2. 미해결 리뷰 코멘트 가져오기: `gh api repos/{owner}/{repo}/pulls/{pr_number}/comments`
3. 각 리뷰를 분석하여 적용 여부 판단
4. 합당한 리뷰는 코드에 반영
5. 합당하지 않은 리뷰는 PR에 거절 사유 코멘트 작성
6. 처리 결과 요약 출력

## 리뷰 가져오기

```bash
# PR 번호 확인
gh pr view --json number -q .number

# 미해결 리뷰 코멘트 가져오기 (resolved 되지 않은 것만)
gh api repos/{owner}/{repo}/pulls/{pr_number}/reviews
gh api repos/{owner}/{repo}/pulls/{pr_number}/comments
```

## 리뷰 판단 기준

### 적용해야 할 리뷰

- **버그 지적**: 명백한 로직 오류, 예외 처리 누락
- **보안 이슈**: 취약점, 민감 정보 노출
- **성능 문제**: 명확한 성능 저하 요인
- **코드 컨벤션**: 프로젝트 규칙 위반
- **가독성 개선**: 명확하게 더 나은 코드 제안

### 거절할 수 있는 리뷰

- **취향 차이**: 동등한 대안 중 단순 선호도
- **과도한 추상화**: 현재 요구사항에 불필요한 복잡도
- **범위 초과**: PR 목적과 무관한 리팩토링 요청
- **근거 부족**: 이유 없이 변경 요청

## 리뷰 적용 방식

1. 리뷰 코멘트의 `path`, `line`, `body` 정보 확인
2. 해당 파일과 라인을 찾아 코드 수정
3. 수정 완료 후 다음 리뷰로 진행
4. 커밋은 사용자가 직접 진행

## 거절 시 코멘트 형식

```bash
gh api repos/{owner}/{repo}/pulls/{pr_number}/comments/{comment_id}/replies \
  -X POST \
  -f body="[거절 사유 설명]"
```

거절 코멘트는 정중하고 구체적인 이유를 포함해야 합니다:

```
감사합니다. 다만 이 부분은 [구체적 이유]로 인해 현재 구현을 유지하려고 합니다.

- [근거 1]
- [근거 2]
```

## 처리 결과 출력

모든 리뷰 처리 후 요약을 출력합니다:

```
## 리뷰 처리 결과

### 적용됨 (N건)
- file.ts:42 - [리뷰 내용 요약]
- file.ts:58 - [리뷰 내용 요약]

### 거절됨 (N건)
- file.ts:100 - [거절 사유 요약]

### 처리 필요
- 커밋되지 않은 변경사항이 있습니다. `git diff`로 확인 후 커밋해주세요.
```
